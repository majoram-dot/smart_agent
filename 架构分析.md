# SmartAgent 架构分析

## 📊 结论：严格分层单向依赖架构 ✅

经过两个阶段的重构，SmartAgent 已从早期的混合式结构演进为**严格分层、单向依赖**的架构设计，彻底消除了循环依赖并建立了清晰的职责边界。

---

## 🏗️ 分层架构图

```
                    ┌─────────────────────────────────┐
                    │      接口层 (UI / CLI)           │
                    │   gradio_ui  ·  cli.py           │
                    └───────────────┬─────────────────┘
                                    │
                    ┌───────────────▼─────────────────┐
                    │      功能扩展层                  │
                    │  deep_research  ·  evaluation   │
                    └───────────────┬─────────────────┘
                                    │
                    ┌───────────────▼─────────────────┐
                    │      协调层 (agents/core.py)     │
                    │   CodeAgent / ToolCallingAgent   │
                    └──────┬────────────────┬──────────┘
                           │                │
          ┌────────────────▼──┐    ┌────────▼────────────────┐
          │    服务层（左侧）  │    │     服务层（右侧）       │
          │  tools  executors  │    │  memory  llm  monitoring │
          │  rag    mcp  utils │    │  serialization  prompts  │
          └────────────────┬──┘    └────────┬────────────────┘
                           │                │
                    ┌──────▼────────────────▼──────────┐
                    │       基础协议层 (base/)           │
                    │   base/tool.py (AbstractTool)     │
                    │   base/schema.py                  │
                    └──────────────────────────────────┘
```

**规则**: 每一层只能向下引用，禁止向上或跨层引用。

---

## 📦 各层详细分析

### 1. `base/` — 基础协议层（无任何业务依赖）

| 文件 | 作用 |
|------|------|
| `tool.py` | 抽象 `Tool` 基类：声明 `__call__`、`run`（兼容字典传参）、`to_tool_calling_prompt` |
| `schema.py` | JSON Schema 类型工具函数 |

**关键设计**:
- `Tool` 为抽象基类（`ABC`），强制要求子类实现 `__call__`
- 提供向后兼容的 `run(dict)` 拦截解包方法，无缝支持旧版 Demo 调用风格

---

### 2. 服务层 — 相互独立，可按需引入

#### `tools/`
| 文件 | 说明 |
|------|------|
| `core.py` | `Tool`, `BaseTool`, `ToolCollection`（继承 `base.tool`） |
| `default.py` | `PythonInterpreterTool`, `FinalAnswerTool`, `TOOL_MAPPING` |
| `validation.py` | `MethodChecker`, `validate_tool_attributes`（AST 工具校验） |
| `note_tool.py` | `NoteTool`（JSON 文件持久化笔记） |
| `terminal_tool.py` | `TerminalTool`（安全白名单命令执行） |
| `web_search.py` | `WebSearchTool`（DuckDuckGo 搜索） |

#### `memory/`
| 文件 | 说明 |
|------|------|
| `core.py` | `AgentMemory`, `ActionStep`, `PlanningStep`, `ToolCall` 等运行时步骤记忆 |
| `memory_manager.py` | `MemoryManager` — 统一管理 4 种长期记忆 |
| `memory_tool.py` | 将 MemoryManager 包装为 Agent 可调用工具 |
| `*_memory.py` | 工作 / 情景 / 语义 / 感知记忆各自独立实现 |

#### `monitoring/`
- `core.py`: `AgentLogger`, `Monitor`, `TokenUsage`, `LogLevel`, `Timing`

#### `executors/`
- `local_python.py`: `LocalPythonInterpreter`, `evaluate_python_code`
- `remote.py`: `DockerExecutor`, `E2BExecutor`, `ModalExecutor`, `BlaxelExecutor`, `WasmExecutor`

#### `llm/`
- `models.py`: `LiteLLMModel`, `OpenAIServerModel`, `AzureOpenAIServerModel`, `TransformersModel`
- `chat.py`: 轻量级 LLM 封装（支持 DashScope / Zhipu / Siliconflow），专为 RAG 场景优化。

#### `rag/`
完全独立的 RAG 子系统，不依赖任何 Agent 或 Tool：
- `vector_store.py`, `embeddings.py`, `reranker.py`, `document_loader.py`
- `pipeline.py`（GSSC / MarkItDown）
- `advanced_search.py`（MQE / HyDE / Hybrid）

---

### 3. `agents/` — 协调层（唯一具横向整合权的模块）

```python
# agents/core.py 合法依赖路径
from ..tools.core import Tool, validate_tool_arguments
from ..tools.default import FinalAnswerTool, TOOL_MAPPING
from ..memory.core import AgentMemory, ActionStep, ...
from ..monitoring.core import AgentLogger, Monitor, ...
from ..executors.local_python import LocalPythonInterpreter
from ..executors.remote import DockerExecutor, ...
from ..llm.models import Model, ChatMessage, ...
```

任何从 `agents` 层反向引用 `tools` 或 `memory` 内部细节的导入路径均列为架构违规。

---

### 4. 功能扩展层

#### `deep_research/`
- 构建在 `agents` 层之上的高层 Agent
- 通过 `from ..agents.core import ToolCallingAgent` 等路径引入协调层能力

#### `evaluation/` 与 `benchmarks/`（外部解耦评测基础设施）
- `src/smart_agent/evaluation/`: 通用指标基类、评估数据集加载逻辑（BFCL/GAIA）、匹配器（AST/QUE精细匹配等）
- `benchmarks/`: 分离式的执行脚本（如 `run_bfcl.py`）及专门依赖隔离（`requirements-eval.txt`），彻底消除评测工具链依赖冲突问题

---

## 🔗 耦合度总结

| 模块 | 耦合度 | 可独立使用 | 说明 |
|------|--------|-----------|------|
| `base` | 🟢 零耦合 | ✅ | 无业务依赖 |
| `llm` | 🟢 极低 | ✅ | 只依赖外部 API |
| `rag` | 🟢 极低 | ✅ | 独立子系统 |
| `memory` | 🟢 极低 | ✅ | 两层互不干扰 |
| `tools` | 🟢 极低 | ✅ | 每个工具独立 |
| `executors` | 🟢 极低 | ✅ | 按需引入 |
| `monitoring` | 🟢 极低 | ✅ | 纯日志封装 |
| `agents` | 🟡 中等 | ❌ | 协调层，需要服务层支撑 |
| `deep_research` | 🟡 中等 | ❌ | 依赖 agents 层 |
| `evaluation` | 🟢 极低 | ✅ | 与核心完全解耦 |
| `ui` | 🟡 中等 | ❌ | 展示层，依赖 agents |

## 💡 典型使用场景

### 场景 1: 仅使用 RAG
```python
from smart_agent.rag import VectorStore, BgeEmbedding, Documents
# 完全不引入 Agent 层
```

### 场景 2: 仅使用工具
```python
from smart_agent.tools import NoteTool, TerminalTool

notes = NoteTool()
notes.run({"action": "create", "title": "测试", "content": "内容"})
```

### 场景 3: 完整 Agent
```python
from smart_agent import CodeAgent, LiteLLMModel, FinalAnswerTool
from smart_agent.tools import WebSearchTool

model = LiteLLMModel(model_id="openai/qwen-plus", api_base="...")
agent = CodeAgent(tools=[WebSearchTool()], model=model)
agent.run("研究量子计算的现状")
```

### 场景 4: 深度研究 Agent
```python
from smart_agent.deep_research import run_deep_research

result = run_deep_research("大语言模型的最新进展", model=model)
```

---

## 🏆 架构优势

1. **零循环依赖**: 通过严格分层，从根本上杜绝了循环引用
2. **按需引入**: 任意子系统可独立安装和使用
3. **扩展友好**: 新功能只需在正确层级上挂载，不影响已有系统
4. **向后兼容**: `base.tool.Tool.run()` 在保持接口稳定的同时拥抱新架构
5. **测试隔离**: `evaluation/` 与 `benchmarks/` 完全独立，不污染主包依赖
